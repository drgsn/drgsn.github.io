Design Patterns Intro

Oct 06, 2021
4 minute read

There is an old saying, “If the only tool you have is a hammer, you will start treating all your problems like a nail.”

This quote dates back to 1964 and Abraham Kaplan, whose first wording was actually:

“Give a small boy a hammer and he will find that everything he encounters needs a pounding.”

I’ve noticed that in Software Engineering interviews there is a big accent on algorithms and data structures, but not on design patterns, at least not when you are starting out. Based on this assumption, I think that we encourage people to learn a lot of algorithms that they might not use (but nevertheless are helpful to know about) and not learn about design patterns, that will come in handy when they need to design and implement solutions to different problems. When they’ll learn a new design pattern, they will start to overuse it, just like a small boy with a hammer or Jeremy Clarkson.

I know that when you start advancing in your career you’ll start to learn about different design patterns, understand why they are important and when to use them, but you shouldn’t hit the wall before you start to look where are you going. You should learn about design patterns before you need them, so when the occasion presents itself, you know which one is a better fit to solve the problem that you need to solve. I don’t remember learning about design patterns during my university years maybe I didn’t pay enough attention, so I’m taking the time now to write about some patterns that I’ve used and about some that I’ve learned, but didn’t have the chance to use, yet. This will help me to better understand them, maybe learn a couple of new ones and hopefully teach you something new.

I will write posts about different patterns that are in general useful in object oriented programming (because that’s what I use in general). I’ve used the Proxy Pattern to do some refactoring and when my changes were put in a PR(pull request) the reviewers were impressed. I know that the person that implemented the solution or one of the reviewers should’ve thought that this pattern was the perfect fit, but what if the implementer didn’t know about it and the reviewers didn’t actually think if the solution was the best one, but was only interested that it works and is cleanly implemented? I’ve been guilty of both of those things before.

Lastly, let’s give a definition for design patterns: Design patterns are template solutions to general problems that software developers try to solve. These solutions were obtained by trial and error and were deemed the best way to solve the problem. Design patterns represent the best practices used by software developers.

From Wikipedia: Design patterns can speed up the development process by providing tested, proven development paradigms. Effective software design requires considering issues that may not become visible until later in the implementation. Freshly written code can often have hidden subtle issues that take time to be detected, issues that sometimes can cause major problems down the road. Reusing design patterns helps to prevent such subtle issues, and it also improves code readability for coders and architects who are familiar with the patterns.

By definition, a pattern must be programmed anew into each application that uses it. Since some authors see this as a step backward from software reuse as provided by components, researchers have worked to turn patterns into components. Meyer and Arnout were able to provide full or partial componentisation of two-thirds of the patterns they attempted.

Types of Design Patterns

As per the design pattern reference book Design Patterns - Elements of Reusable Object-Oriented Software, there are 23 design patterns which can be classified in three categories: Creational, Structural and Behavioral patterns.

# Pattern & Description
1. Creational Patterns - These design patterns provide a way to create objects while hiding the creation logic, rather than instantiating objects directly using new operator. This gives program more flexibility in deciding which objects need to be created for a given use case.
2. Structural Patterns - These design patterns concern class and object composition. Concept of inheritance is used to compose interfaces and define ways to compose objects to obtain new functionalities.
3. Behavioral Patterns - These design patterns are specifically concerned with communication between objects.
