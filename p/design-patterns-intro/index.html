Design Patterns Intro

Oct 06, 2021
3 minute read

There is an old saying, “If the only tool you have is a hammer, you will start treating all your problems like a nail.”

This quote dates back to 1964 and Abraham Kaplan, whose first wording was actually:

“Give a small boy a hammer and he will find that everything he encounters needs a pounding.”

I’ve noticed that in Software Engineering interviews there is a big accent on algorithms and data structures, but not on design patterns, at least not when you are starting out. Based on this assumption, I think that we encourage people to learn a lot of algorithms that they might not use (but nevertheless might be helpful to know about) and not learn about different design patterns, that will come in handy when they need to design different solutions. When they’ll learn a new design pattern, they will start to overuse it, just like a small boy with a hammer or Jeremy Clarkson.

I know that when you start advacing in your career you’ll start to learn different design patterns, understand why they are important and when to use them, but you shouldn’t hit the wall before you start to see. You should learn about these patterns before you actually need them, so when the ocassion presents itself, you know which one to use. I don’t remember to been thaught about design patterns during my university years, so I’m taking the time know to write about some patterns that I’ve used and about some that I’ve learned, but didn’t have the chance to use.

I will write posts about different patterns that are in general usefull in object oriented programming(because that’s what I’ve mostly used) and I’m starting this series with the Proxy Pattern. I’ve used it recently at work when I needed to do some refactoring and when it was put in a PR(pull request) the reviewers were impresent. I know either the person that implemented to solution or one of the reviewers should’ve thought that this pattern was the perfect fit, but what if the implementer didn’t know about it and the reviers didn’t actually think if the solution was the best one, but were only interested that it works and is cleanly implemented? I’ve been guilty of both of those things before.

Lastly, lets give a definition for design patterns: Design patterns represent the best practices used by software developers. Design patterns are template solutions to general problems that software developers try to solve. These solutions were obtained by trial and error and were deemed the best way to solve the problem

From Wikipedia: Design patterns can speed up the development process by providing tested, proven development paradigms. Effective software design requires considering issues that may not become visible until later in the implementation. Freshly written code can often have hidden subtle issues that take time to be detected, issues that sometimes can cause major problems down the road. Reusing design patterns helps to prevent such subtle issues, and it also improves code readability for coders and architects who are familiar with the patterns.

By definition, a pattern must be programmed anew into each application that uses it. Since some authors see this as a step backward from software reuse as provided by components, researchers have worked to turn patterns into components. Meyer and Arnout were able to provide full or partial componentization of two-thirds of the patterns they attempted.
