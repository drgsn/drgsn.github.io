Factory Pattern

Oct 12, 2021
5 minute read

Factory pattern or Factory Method Pattern is one of the most basic and most used design patterns in OOP languages. This is a creational type pattern and this pattern provides one of the best ways to create different objects without exposing the creation logic and exposing it using a common interface.

To make it easier to understand, think of a car factory. The car factory produces multiple car types, but you don’t know exactly how the cars are produced, how much of the underlying technology and components they share, but you don’t care about that.

When you chose a car, you make the choice based on what it offers you, not on what it shares with other cars, that’s the problem that the manufacturer needs to figure out. You are interested just in getting the select model you choose from the factory.

In other words it provides an interface for creating families of related or dependent objects without specifying their concrete classes.

Implementation

Keeping the previous analogy in mind, let’s look at a diagram of how we would use the factory pattern.

We’re going to create a Car interface and a few classes implementing the Car interface. A factory class CarFactory is defined as the next step.

FactoryPatternDemo, our demo class will use CarFactory to get a Car object. It will pass information (ModelA / ModelB / ModelC) to CarFactory to get the type of object it needs.

Step 1

Create an interface.

Car.java

public interface Car {
   void drive();
}

Step 2

Create concrete classes implementing the same interface.

ModelA.java

public class ModelA implements Car {

   @Override
   public void drive() {
      System.out.println("Inside ModelA::drive() method.");
   }
}

ModelB.java

public class ModelB implements Car {

   @Override
   public void drive() {
      System.out.println("Inside ModelB::drive() method.");
   }
}

Model.java

public class ModelC implements Car {

   @Override
   public void drive() {
      System.out.println("Inside ModelC::drive() method.");
   }
}

Step 3

Create a Factory to generate an object of concrete class based on given information.

CarFactory.java

public class CarFactory {

   //use getCarModel method to get object of type Car 
   public Car getCarModel(String carModel){
      if(carModel == null){
         return null;
      } 
      if(carModel.equalsIgnoreCase("ModelA")){
         return new ModelA();
   
      } else if(carModel.equalsIgnoreCase("ModelB")){
         return new ModelB();
   
      } else if(carModel.equalsIgnoreCase("ModelC")){
         return new ModelC();
      }
  
      return null;
   }
}

Step 4

Use the Factory to get the objects of the concrete class by passing information such as type.

FactoryPatternDemo.java

public class FactoryPatternDemo {

   public static void main(String[] args) {
      CarFactory carFactory = new CarFactory();

      //get an object of ModelA and call its drive method.
      Car modelA = carFactory.getCarModel("ModelA");

      //call drive method of ModelA
      modelA.drive();

      //get an object of ModelB and call its drive method.
      Car modelB = carFactory.getCarModel("ModelB");

      //call drive method of ModelB
      modelB.drive();

      //get an object of ModelC and call its drive method.
      Car modelC = carFactory.getCarModel("ModelC");

      //call drive method of ModelC
      modelC.drive();
   }
}

Use cases

  • A class cannot anticipate the type of objects it needs to create beforehand.

    • The Factory Pattern separates the object construction code from the code that uses the object. Therefore it’s easier to extend the object construction code independently from the rest of the code. For example, to add a new object type to the app, you’ll only need to create a new creator subclass and override the factory method in it.

  • A class requires its subclasses to specify the objects it creates.

  • You want to localize the logic to instantiate a complex object. For example the connection to a database. Let’s think about what has to be done to reuse an existing connection to the database:

    First, you need to create some storage to keep track of all of the opened connections.

    When someone wants to interact with the database, the program should look for a free connection inside that pool, and then return it to the client code.

    If there are no free connection, the program should create a new one and add it to the pool. That’s a lot of code! And it must all be put into a single place so that you don’t pollute the program with duplicate code.

    Probably the most obvious and convenient place where this code could be placed is in the constructor of the class whose objects we’re trying to reuse. However, a constructor must always return new objects by definition. It can’t return existing instances.

    Therefore, you need to have a regular method capable of creating new objects as well as reusing existing ones. That sounds very much like a factory method.

The Factory Method Pattern is useful when you need to abstract the creation of an object away from its actual implementation. Let’s say the factory will be building a “Car” product type. A car could be made up of any number of components, some of which can and will change later, depending on advances in technology.

Pros and Cons

Pros:

  • You avoid tight coupling between the creator and the created objects.
  • Single Responsibility Principle. You can move the product creation code into one place in the program, making the code easier to support.
  • Open/Closed Principle. You can introduce new types of products into the program without breaking the existing client code.

Cons:

  • The code may become more complicated since you need to introduce a lot of new subclasses to implement the pattern. The best-case scenario is when you’re introducing the pattern into an existing hierarchy of creator classes.
